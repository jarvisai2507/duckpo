## 왜 이 런북이 필요한가 (고장 유형)

웹/원격 Claude 세션에서 Supabase(금고)에 닿지 못하면 **브리핑도, 기록 저장도** 안 된다. 원인은 셋 중 하나다:

| # | 고장 | 증상 | 누가 고치나 |
|---|------|------|------------|
| A | 환경 네트워크 정책이 Supabase 호스트 차단 | REST/커넥터가 `403 policy denial` | **관리자** (환경 설정) |
| B | Supabase MCP 커넥터 미부착 | 세션에 `execute_sql` 도구 없음 | **관리자** (커넥터 부착) |
| C | 무료 플랜 자동 정지(약 1주 무활동) | 모든 요청 실패/타임아웃 | 자동 예방(워크플로) + 대시보드 원클릭 복구 |

> 대통령은 A·B를 알 수도, 고칠 수도 없다. 세션에서 장애가 감지되면 비서실장이 **관리자용 조치 항목**으로 남긴다(무손실 폴백은 `CLAUDE.md`의 "🔌 커넥터/네트워크 장애 대응 프로토콜").

## 1. 라이브 연결 세팅 (근본 복구 · A·B)

**실측으로 확정된 현재 상태**: 이 환경의 **Network access = Trusted(기본값)** — GitHub·패키지저장소·클라우드SDK만 허용하고 **Supabase를 포함한 그 외 모든 도메인은 게이트웨이가 403으로 차단**한다. (google.com·example.com도 동일하게 차단됨을 확인.) 커넥터 "부착" 토글과 별개의, 환경 **네트워크 접근 레벨** 문제다.
문서: https://code.claude.com/docs/en/claude-code-on-the-web#network-access

라이브 연결에는 아래 **둘 중 하나(또는 둘 다)** 를 세팅한다.

### 방법 1 — Supabase MCP 커넥터 활성화 (`execute_sql`용 · 권장)
- **MCP 커넥터 트래픽은 Anthropic 서버를 경유**하므로 Allowed domains에 넣지 않아도 동작한다(문서 명시). 네트워크 레벨과 무관하게 `execute_sql`을 쓸 수 있다.
- 커넥터는 **세션 또는 루틴(routine) 단위로** 설정한다 → 클라우드 세션/루틴 설정에서 Supabase 커넥터를 켠다.

### 방법 2 — Network access를 Custom/Full로 (직접 REST/curl용)
- 환경 편집(클라우드 아이콘 → 편집 다이얼로그) → **Network access** 선택기. (별도 Environments 페이지는 없다.)
- **Custom** 선택 → 나타나는 **Allowed domains** 칸에 한 줄씩 입력:
  ```
  *.supabase.co
  ```
  "Also include default list of common package managers" 체크 유지(GitHub 등 보존). 또는 **Full**(모든 도메인) 선택.

### (선택) 민원 사이트 자동화용 도메인 허용
대통령이 원하는 작업: 세션 안 브라우저(Chromium+Playwright 내장)로 **정보공개포털 청구서 일괄 다운로드 + 증거 체계 자동 매핑**.
2026-07-06 실측 — 민원 사이트는 게이트웨이 403 차단 상태라 현재 불가(참고: `*.supabase.co`는 이미 개방 확인됨). 허용하려면 Allowed domains에 추가:
  ```
  *.open.go.kr

  *.bai.go.kr

  ```

## 2. 무료 플랜 자동 정지 예방·복구 (고장 C)

- **자동 예방**: `.github/workflows/keepalive-supabase.yml` 이 매일 공개 RPC를 1회 호출해
  DB 활동을 만들어 자동 정지를 막는다. (GitHub 러너는 네트워크 제한 없음.)
- **상태 확인**: GitHub → Actions → "Keep Supabase alive" 가 **초록**이면 정상.
  빨간 X면 프로젝트 정지 또는 설정 변경 신호다.
- **정지됐을 때 복구**: Supabase 대시보드에서 프로젝트 **Restore(재개)** 원클릭. 데이터는 보존된다.
- **주의(GitHub 정책)**: 저장소가 60일간 커밋 등 활동이 전혀 없으면 GitHub가 예약 워크플로를
  비활성화할 수 있다. 가끔 커밋이 있으면 유지되며, 필요 시 Actions에서 워크플로를 수동 실행(Run workflow)해 되살린다.

## 3-1. 체계 분리 원칙 (코워크 ↔ 코드) — 2026-07-12 감사·명문화

두 체계는 **설계상 분리**돼 있으며, 이 원칙은 불변이다. (관리자 감사 결과 위반 0건 확인)

| 구분 | 코워크(Cowork) | 코드(비서실장 원격 세션) |
|---|---|---|
| 위치 | 대통령 PC의 Claude Desktop | 클라우드 원격 실행 환경 |
| 권한 | 로컬 디스크(E:\클로더)·로그인된 브라우저·민원사이트 수집 | **금고(Supabase) 쓰기·엣지함수 배포·GitHub 단독 권한** |
| 이 세션 접속 | **불가** (접속 수단 없음) | 해당 |

- **교량은 대통령 손수 운반뿐**: 코워크↔코드 간 자동 실행 경로는 없다. 코워크 산출물(보고문·목록·zip·분할조각)은 **대통령이 붙여넣기/파일 드래그로** 이 세션에 전달할 때만 들어온다.
- **불변 규칙**:
  1. 임시 반입구(uploader) 비밀열쇠는 **이 세션 내에서 생성→사용→410 재봉쇄**. 코워크에 보내는 지시문에 **열쇠·service_role·접속정보를 넣지 않는다.**
  2. 코워크가 전달한 반입물은 **바이트 크기·SHA256을 이 세션이 독립 검증**한 뒤에만 금고에 반영한다(대형본은 조각 재조립 후 지문 재검증).
  3. 코워크의 진단 주장(미반입 건수·오류 등)은 **금고 직접 조회로 재확인**한 뒤에만 조치한다.
  4. 코워크는 자기 샌드박스에서 Supabase 직접 접근이 차단(HTTP 000)돼 있어야 정상이다. 익명 쓰기 경로를 여는 요청은 거부한다.
- **감사 근거(2026-07-12)**: 권한 직접사용 0 / 열쇠 월경 0 / GitHub 우회저장 0(문서만) / 반입구 매회 410 봉쇄 / 원본(case_evidence 기존 행) 무단변경 0.
